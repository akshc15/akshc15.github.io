DOCKER · MISSION 01 ⚡ +50 XP
Your First Container
Run hello-world. Understand images vs containers. See Docker pull, create, start in real-time.
🍕
Image vs Container — The Pizza Analogy
Docker Image = frozen pizza in a box. Recipe + ingredients, all sealed. You can make 10 pizzas from 1 box.

Docker Container = pizza in the oven, running. When you docker run, Docker takes the frozen image and starts it as a live container.

Docker Hub = the supermarket. Millions of pre-made images (nginx, node, ubuntu, postgres) that you can pull for free.
1
Run hello-world on your GCP VM
todo
Open your GCP VM terminal in VS Code → run this command:
forge@gcp-vm:~$
docker run hello-world

# Expected output:
Hello from Docker!
This message shows that your Docker installation appears to be working correctly.

# What Docker did behind the scenes:
# 1. Looked for hello-world image locally → not found
# 2. Pulled it from Docker Hub → downloaded
# 3. Created container from image
# 4. Started container → it printed message → exited
✅ Perfect!
Docker is working on your VM. The image was automatically pulled from Docker Hub. Notice how small it is — hello-world is only 13KB!
2
Inspect what Docker created
todo
These 3 commands are your daily Docker toolkit — you'll use them 50 times a day:
Daily toolkit
# Running containers (live right now)
docker ps
CONTAINER ID IMAGE STATUS PORTS NAMES
# Empty — hello-world finished and exited

# ALL containers including stopped ones
docker ps -a
abc123def hello-world Exited (0) 2m ago loving_tesla

# All local images (your library)
docker images
REPOSITORY TAG IMAGE ID SIZE
hello-world latest d1165f2212 13.3kB
✅ Understanding the output
Exited (0) = finished successfully (exit code 0 = success in Linux, non-zero = error). loving_tesla = Docker auto-generates fun names if you don't set --name. IMAGE ID = unique fingerprint of the image.
3
Run a real web server — nginx
todo
Let's run a container that stays running — an nginx web server:
Run nginx web server
# -d = detached (runs in background)
# -p 8080:80 = VM port 8080 → container port 80
# --name = give it a friendly name
docker run -d -p 8080:80 --name my-web nginx

# It's running!
docker ps
CONTAINER ID IMAGE STATUS PORTS
xyz789abc nginx Up 3s 0.0.0.0:8080->80/tcp

# Test it
curl http://localhost:8080
# Returns: "Welcome to nginx!"

# See its logs
docker logs my-web

# Stop and remove when done
docker stop my-web && docker rm my-web
🎉 You ran your first real service in Docker!
nginx is a production web server used by Netflix, GitHub, Cloudflare. You just ran it in 1 command with zero installation. That's the power of containers.
What is the difference between docker stop and docker rm?
🚨 Errors You'll Hit + Fixes
Got permission denied while trying to connect to the Docker daemon
✅ FIX: sudo usermod -aG docker $USER → then logout and login again (or: newgrp docker)
docker: Error response from daemon: port is already allocated
✅ FIX: Something else uses port 8080. Use a different port: -p 8081:80 OR kill the process: lsof -i :8080 → kill -9 PID
Unable to find image 'hello-world:latest' locally — then network timeout
✅ FIX: No internet on VM. Check: curl google.com. If fails, check GCP firewall rules allow egress traffic.
DOCKER · MISSION 02 ⚡ +80 XP
Write Your First Dockerfile
Package DreamChat backend into a Docker image. Multi-stage build. Security best practices from day one.
📋
Dockerfile = Recipe for your Image
Each line in a Dockerfile creates a layer. Layers are cached — if you only change your code (not package.json), Docker reuses the npm install layer. This makes rebuilds 10× faster.

Order matters! Put things that change least at top (FROM, npm install), things that change most at bottom (COPY . .). This maximizes cache hits.
1
Create the DreamChat backend
todo
Create project
mkdir -p ~/dream-chat/backend && cd ~/dream-chat/backend

# Create simple server
cat > server.js << 'EOF'
const http = require('http');
const server = http.createServer((req, res) => {
  if (req.url === '/health') {
    res.writeHead(200);
    res.end(JSON.stringify({status:'ok',uptime:process.uptime()}));
  } else {
    res.end('DreamChat Backend 🚀');
  }
});
server.listen(3001, () => console.log('Running on 3001'));
EOF

npm init -y
✅ Good
You have server.js and package.json. Notice we added a /health endpoint — Kubernetes will use this to check if your app is alive!
2
Write the production-grade Dockerfile
todo
backend/Dockerfile
nano Dockerfile

# ═══ Paste this exactly ═══

# Stage 1: Install dependencies (build stage)
FROM node:20-alpine AS builder
WORKDIR /app
# Copy package files FIRST (cache optimization!)
COPY package*.json ./
RUN npm ci --only=production

# Stage 2: Runtime (final tiny image)
FROM node:20-alpine
# Security: never run as root
RUN addgroup -S appgrp && adduser -S appuser -G appgrp
WORKDIR /app
# Copy only what we need from builder
COPY --from=builder /app/node_modules ./node_modules
COPY . .
# Switch to non-root user
USER appuser
EXPOSE 3001
# Health check — K8s uses this!
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -q http://localhost:3001/health || exit 1
CMD ["node", "server.js"]
✅ Why each line is important
node:20-alpine = 5MB base (vs 900MB ubuntu). npm ci = reproducible installs (locks versions). --only=production = skip devDependencies. Multi-stage = build tools not in final image. USER appuser = security. HEALTHCHECK = K8s restarts unhealthy pods.
3
Build, run, and verify
todo
Build + run your image
# Build the image (. = current directory)
docker build -t dream-backend:v1 .
Step 1/10 : FROM node:20-alpine AS builder
Successfully built a1b2c3d4e5f6
Successfully tagged dream-backend:v1

# Check image size
docker images dream-backend
dream-backend v1 abc123 ~85MB ← tiny!

# Run it
docker run -d -p 3001:3001 --name backend dream-backend:v1

# Test health endpoint
curl http://localhost:3001/health
{"status":"ok","uptime":2.34} ←
🎉 You built a production-grade Docker image!
~85MB image. Non-root user. Health check. Multi-stage build. This is better than 80% of Dockerfiles in the wild. You just went from code to containerized app in minutes.
💡
Interview tip: When asked "how do you optimize a Dockerfile?" say: multi-stage builds, alpine base image, order layers by change frequency (least-changed first), combine RUN commands with &&, use .dockerignore to exclude node_modules and .git from build context.
🚨 Common Dockerfile Errors + Fixes
COPY failed: file not found in build context: stat package.json
✅ FIX: You're running docker build from wrong directory. Must be in the folder with Dockerfile: cd ~/dream-chat/backend && docker build -t dream-backend:v1 .
RUN npm ci → sh: npm: not found
✅ FIX: Alpine images use minimal shell. The issue is the WORKDIR might not exist. Add: RUN mkdir -p /app before WORKDIR /app
container exited immediately with code 1
✅ FIX: Check logs: docker logs backend. 99% of the time it's an app error — missing env var, can't connect to DB, syntax error in code.

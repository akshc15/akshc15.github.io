Docker · Mission 01 ⚡ +50 XP

Your First Container

Run hello-world. See what Docker actually does behind the scenes. Understand images vs containers forever.

🍕
Think of it like a frozen pizza
Docker Image = frozen pizza in a box. Recipe + ingredients sealed together. You can make 10 pizzas from 1 box.

Docker Container = pizza in the oven, running hot. When you run docker run, Docker takes the frozen image and starts it as a live container.

Docker Hub = the supermarket. Millions of free pre-made images — nginx, node, ubuntu, mongodb.
1
Run hello-world on your cloud VM
Open your cloud VM terminal in VS Code and run this command:
cloud-vm ~ docker
# Pull and run hello-world
you@vm:~$ docker run hello-world

# Expected output:
Hello from Docker!
This message shows Docker is working correctly.
✅ What happened
Docker checked locally — no hello-world image found. Pulled it from Docker Hub automatically. Created a container, ran it, it printed the message, then exited. That's it! The image is now saved locally for next time.
2
Inspect what Docker created
These 3 commands are your daily Docker toolkit:
daily toolkit
# See running containers
you@vm:~$ docker ps
CONTAINER ID IMAGE STATUS PORTS
# Empty — hello-world already exited

# See ALL containers including stopped
you@vm:~$ docker ps -a
abc123 hello-world Exited(0) 2m ago

# See all local images
you@vm:~$ docker images
hello-world latest 13.3kB
✅ What each means
Exited (0) = finished successfully. Exit code 0 = success in Linux. Non-zero = error. 13.3kB = tiny image! Compare: Ubuntu is 70MB, node is 900MB.
3
Run nginx — a real web server
Run a container that stays running. nginx serves web pages to millions of users at Netflix, GitHub, Cloudflare.
run nginx
# -d = run in background
# -p 8080:80 = Mac port 8080 → container port 80
# --name = friendly name
you@vm:~$ docker run -d -p 8080:80 --name web nginx

# Test it
you@vm:~$ curl http://localhost:8080
Welcome to nginx! ✓

# Clean up
you@vm:~$ docker stop web && docker rm web
🎉 You ran your first real service!
nginx is running inside a container on your cloud VM. You accessed it with curl. 1 command = running web server. No apt-get install. No config files. Just docker run.
What is the difference between docker stop and docker rm ?
🚨
Common errors you'll hit
Got permission denied while trying to connect to the Docker daemon
Fix: sudo usermod -aG docker $USER then log out and log back in to your VM. Or run: newgrp docker
docker: Error response: port is already allocated
Fix: Something else uses port 8080. Use different port: -p 8081:80 Or find and kill: lsof -i :8080 then kill -9 PID
Docker · Mission 02 ⚡ +80 XP

Write Your First Dockerfile

Package DreamChat backend into a Docker image. Multi-stage build. Security best practices from day one.

📋
Dockerfile = Recipe for your image
Each line creates a layer. Layers are cached — if you only change your code (not package.json), Docker reuses the npm install layer. Builds go from 45 seconds to 8 seconds.

Order matters! Put things that change least at top. Things that change most at bottom. This maximises cache hits every time.
1
Create the DreamChat backend
create project
you@vm:~$ mkdir -p ~/dream-chat/backend
you@vm:~$ cd ~/dream-chat/backend

# Create simple server
you@vm:~$ cat > server.js << 'EOF'
const http = require('http');
const server = http.createServer((req, res) => {
if (req.url === '/health') {
res.end(JSON.stringify({status:'ok'}));
} else { res.end('DreamChat 🚀'); }
});
server.listen(3001,()=>console.log('Running!'));
EOF

you@vm:~$ npm init -y
✅ Good
You have server.js and package.json. Notice the /health endpoint — Kubernetes will use this to check if your app is alive. Add it to every app you build.
2
Write the production Dockerfile
backend/Dockerfile
you@vm:~$ nano Dockerfile

# Paste this:

# Stage 1: Install dependencies
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# Stage 2: Runtime (tiny final image)
FROM node:20-alpine
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
USER app
EXPOSE 3001
CMD ["node", "server.js"]

# Ctrl+X → Y → Enter to save
✅ Why each line matters
node:20-alpine = 5MB base (not 900MB ubuntu!). npm ci = reproducible installs. Multi-stage = build tools NOT in final image. USER app = never run as root.
3
Build and run it
build + run
# Build (. = current folder)
you@vm:~$ docker build -t dream-backend:v1 .
Successfully built abc123
Successfully tagged dream-backend:v1

# Check image size
you@vm:~$ docker images dream-backend
dream-backend v1 ~85MB ← tiny!

# Run and test
you@vm:~$ docker run -d -p 3001:3001 --name backend dream-backend:v1
you@vm:~$ curl http://localhost:3001/health
{"status":"ok"} ✓
🎉 You built a production Docker image!
85MB image. Non-root user. Multi-stage build. Health check endpoint. This is better than 80% of Dockerfiles in the wild. You just containerized an app the professional way.

Docker #01
Easy
🐳 Containers
What is the difference between a Docker IMAGE and a Docker CONTAINER?
✅ Correct! +10 points!
❌ Not quite. The correct answer is B.
💡 Perfect Answer for Interview
Image = frozen snapshot of your app + everything it needs (OS, runtime, code, dependencies). It's built once with docker build and stored. It's read-only.

Container = running image. When you run docker run, Docker takes the image and adds a writable layer on top — that's the container. You can run 10 containers from 1 image simultaneously, each isolated.

Analogy: Image = class definition in code. Container = instance of that class.
Docker #02
Medium
🔨 Dockerfile
Why should you use multi-stage builds in a Dockerfile? What problem does it solve?
✅ Model Answer
Problem: Build tools (compilers, npm, maven) are huge but NOT needed at runtime. Without multi-stage, your final image contains all build tools = bloated image with more attack surface.

Solution — Multi-stage: Stage 1 (builder) installs everything and builds. Stage 2 (runtime) copies ONLY the built artifact — no build tools. Result: image 5–10× smaller, faster to pull, fewer CVEs.

Real numbers from DreamChat: Backend without multi-stage: 890MB. With multi-stage: 85MB. That's 10× smaller!
FROM node:20 AS builder → npm install + build
FROM node:20-alpine → COPY --from=builder /app/dist . → only the output!
Docker #03
Medium
🔒 Security
What happens if you don't specify a USER in your Dockerfile? Why is this a security risk?
✅ Exactly right! Security is critical in interviews!
❌ Correct answer is A.
💡 Key Security Point
Default Docker containers run as root (UID 0). If an attacker exploits your app, they get root inside the container. With container escape vulnerabilities, this can lead to root on the HOST system — catastrophic.

Fix: Always add to Dockerfile:
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
Docker #04
Hard
🌐 Networking
In Docker Compose, Container A tries to connect to Container B using http://localhost:3001 but fails. What's wrong and how do you fix it?
✅ Classic interview trap — you nailed it!
❌ Answer is B — very common interview question!
💡 Container Networking Concept
Each container has its own loopback interface. localhost inside Container A points to Container A itself — not to Container B.

Docker Compose creates a shared network. Containers can reach each other by service name. If your backend service is named backend in docker-compose.yml, use http://backend:3001 from frontend.

This is why in DreamChat we set: REACT_APP_BACKEND_URL=http://backend:3001 in docker-compose, but use the external K8s Service URL in production.
Docker #05
Easy
💾 Volumes
You run MongoDB in a Docker container without volumes. You stop and restart the container. What happens to your data?
✅ Right! Volumes are essential for stateful apps!
❌ Answer is C — data is lost without volumes!
💡 Volumes Are Critical
Containers are ephemeral by default. The writable layer is destroyed when the container is removed (docker rm). Data written inside the container disappears.

Fix — use a named volume:
volumes: - mongo_data:/data/db in docker-compose

Named volumes are stored on the HOST filesystem, outside the container lifecycle. This is how MongoDB, PostgreSQL, Redis survive container restarts/updates.
K8s #01
Easy
☸️ Core Concepts
Your backend pod has CrashLoopBackOff status. What are the first 2 commands you run to debug it?
✅ Perfect debugging flow!
❌ Answer is B — always check logs first!
💡 K8s Debug Flow
Step 1: kubectl logs POD_NAME -n NAMESPACE — see what the app printed before crashing. 90% of issues are here.

Step 2: kubectl describe pod POD_NAME -n NAMESPACE — scroll to Events section at the bottom. Shows: image pull failures, OOMKilled, scheduling issues.

Step 3 (if app still running): kubectl exec -it POD_NAME -- sh — shell inside the container for live debugging.

For CrashLoopBackOff specifically: kubectl logs POD_NAME --previous gets logs from the PREVIOUS crashed instance.
K8s #02
Medium
⚖️ Services
What is the difference between ClusterIP, NodePort, and LoadBalancer service types in Kubernetes?
✅ Complete Answer
ClusterIP (default): Internal only. Only pods inside the cluster can reach this service. Use for: backend service that frontend pods call. NOT exposed to internet.

NodePort: Opens a port (30000–32767) on every node. External traffic can hit ANY_NODE_IP:NodePort. Use for: dev/testing. Not production — exposes raw ports.

LoadBalancer: Provisions a cloud load balancer (AWS ALB, GCP GLB). Gets a real public IP. Routes external traffic to your pods. Use for: production internet-facing services.

In DreamChat: backend-svc = ClusterIP (only frontend pods call it). frontend-svc = LoadBalancer (users hit this from internet).
K8s #03
Hard
🔌 WebSockets
In DreamChat, WebSocket connections drop every 60 seconds in K8s. No such issue on local Docker. What's the likely cause?
✅ Senior-level answer! WebSocket in K8s is a real interview topic!
❌ Answer is A — this is a very common real-world issue!
💡 WebSocket + K8s — Real Production Issue
Two problems with WebSockets in K8s:

1. Load Balancer Timeout: AWS ALB default = 60s idle timeout. WebSocket connections look "idle" between messages. Fix: service.beta.kubernetes.io/aws-load-balancer-connection-idle-timeout: "3600"

2. Session Affinity: Multiple backend pods each maintain their own socket list. If user connects to Pod A but next request goes to Pod B (no such connection) → message lost. Fix: sessionAffinity: ClientIP in Service OR use Redis Pub/Sub to share socket state across pods (production solution).
K8s #04
Medium
📈 Scaling
What is HPA (Horizontal Pod Autoscaler)? How does it decide when to scale up?
✅ Excellent! The formula impresses interviewers!
❌ Answer is C — HPA scales PODS not nodes!
💡 HPA Deep Dive
HPA watches metrics every 15 seconds. Uses: desiredReplicas = ceil(currentReplicas × (actualMetric / targetMetric))

Example: 2 pods, CPU at 90%, target 70%: ceil(2 × 90/70) = ceil(2.57) = 3 → scales to 3 pods.

Important distinctions: HPA = pod scaler. Cluster Autoscaler = node scaler (adds EC2s). KEDA = scales on any metric including queue depth, HTTP requests, custom — used in DreamChat project for scaling on active WebSocket connections.
K8s #05
Easy
🔐 Secrets
You need to store the MongoDB connection string in Kubernetes. Should you put it in a ConfigMap or a Secret? Why?
✅ Security best practice — correct!
❌ Answer is B — always use Secrets for credentials!
💡 ConfigMap vs Secret
ConfigMap: Non-sensitive config. App port, feature flags, log level. Stored as plaintext — anyone with kubectl access can read it.

Secret: Sensitive data. Passwords, API keys, certs. Base64-encoded (note: NOT encrypted by default!). Can enable encryption at rest in K8s config. RBAC can restrict who can read Secrets.

Production best practice: Use HashiCorp Vault or AWS Secrets Manager instead of K8s Secrets. K8s Secrets are only base64, not truly secure. Vault provides real encryption + audit logs.
TF #01
Easy
🏗️ State
What is Terraform state and why is it dangerous to delete the state file?
✅ Model Answer
Terraform state is a JSON file (terraform.tfstate) that maps your .tf code to REAL cloud resources. It remembers: "this aws_instance.jenkins in my code = EC2 i-0abc123 in AWS".

If you delete it: Terraform has no idea what exists. If you run terraform apply again, it thinks nothing exists → creates DUPLICATE resources → you get double EC2s, double costs, potential conflicts.

Best practice: Store in remote backend (S3 + DynamoDB or GCS). Never commit to Git. Enable versioning on the S3 bucket so you can recover old state.
TF #02
Medium
🔄 Commands
What is the correct order of Terraform commands to deploy infrastructure from scratch?
✅ Init → Plan → Apply — always!
❌ Answer is C — Init downloads providers first!
💡 Terraform Workflow
1. terraform init — downloads providers (AWS, GCP plugin), initializes backend. Run ONCE per project.
2. terraform validate — checks syntax errors (optional but good habit).
3. terraform plan — shows what WILL happen. Read this carefully before applying! Shows +created, ~modified, -destroyed.
4. terraform apply — actually creates/modifies/destroys. Type 'yes' to confirm.
5. terraform destroy — removes everything. Critical for cost saving!
TF #03
Hard
🔧 Modules
What is a Terraform module and why would you use one instead of writing all resources in main.tf?
✅ Model Answer
A Terraform module is a reusable package of related resources grouped together. It's a folder with .tf files that you call like a function.

Why use modules:
1. Reusability — Create VPC once, reuse for dev/staging/prod with different variables
2. Abstraction — Team uses module "vpc" {...} without knowing the 20+ resources inside
3. Consistency — Same module = same configuration everywhere. No accidental differences between environments

Example: In DreamChat, we have a modules/gke-cluster module. We call it 3 times with different vars for dev/staging/prod — same K8s config, zero code duplication.
CI/CD #01
Easy
⚙️ Concepts
A developer pushes code. In a CI/CD pipeline, what is the difference between what happens in CI vs CD?
✅ Textbook answer!
❌ Answer is B — classic definition question!
💡 CI vs CD Explained
CI (Continuous Integration): Every code push → auto: checkout code, install deps, run tests, run security scan (Trivy/OWASP), build Docker image. Goal: catch bugs FAST before they reach production.

CD (Continuous Delivery): Passing CI build → auto: push image to registry, update K8s manifest, ArgoCD deploys to staging. May require manual approval for production.

Continuous Deployment = CD without manual approval — every passing build goes to production automatically. Requires very high test confidence.

Linux #01
Easy
🔐 Permissions
What does chmod 400 mykey.pem do and why is it needed for SSH?
✅ File permissions mastery!
❌ Answer is B — permissions are critical for SSH!
💡 Linux Permissions Decoded
chmod 400 = r-------- = owner can READ only, group has nothing, others have nothing.

Permission breakdown: 4=read, 2=write, 1=execute. chmod XYZ where X=owner, Y=group, Z=others.

SSH checks key file permissions before using it. If group/others can read it → SSH says: "UNPROTECTED PRIVATE KEY FILE!" and refuses to connect. This is a security feature — private keys shouldn't be readable by others on shared systems.
Linux #02
Medium
🔍 Process Debug
Your Node.js app is running but not responding on port 3001. What Linux commands do you run to investigate?
✅ Debug Flow
ss -tlnp | grep 3001 — is anything actually listening on 3001? (ss is modern netstat)
ps aux | grep node — is the Node process running?
lsof -i :3001 — what process owns port 3001?
journalctl -u your-service -f — view systemd service logs
curl -v http://localhost:3001/health — test health endpoint directly
ufw status — is firewall blocking the port?

Most common cause: app crashed silently, port is not actually open. ss -tlnp confirms this instantly.
Linux #03
Hard
⚡ Performance
Your VM suddenly becomes slow. CPU is at 100%. How do you find which process is causing it?
✅ Perfect! htop is your best friend!
❌ Answer is A — top/htop for CPU, df for disk!
💡 Performance Toolkit
htop — interactive, sort by CPU (click %CPU header). F9 to kill a process.
ps aux --sort=-%cpu | head -10 — top 10 CPU consumers snapshot
pidstat 1 — per-process CPU every 1 second
strace -p PID — what system calls is the process making? (advanced)

Common culprits: runaway build process, crypto miner (if VM was compromised!), Node.js infinite loop, log rotation gone wrong.
Git #01
Medium
🌿 Branching
What is the difference between `git merge` and `git rebase`? When would you use each in a real team?
✅ Model Answer
git merge — creates a merge commit, preserves full history of when branches diverged. Non-destructive. Safe for shared/public branches.
Use when: merging feature branch to main (via PR). Never loses history.

git rebase — moves your commits on top of the target branch. Creates linear, clean history. Rewrites commit hashes.
Use when: updating your PRIVATE feature branch with latest main (git rebase main). Makes your changes appear to be written AFTER the latest main commits = cleaner PR diff.

Golden Rule: NEVER rebase public/shared branches. Rewriting hashes confuses everyone else who has the old commits.

Real team workflow: git rebase main to update your feature branch → push → PR → maintainer merges with --squash for one clean commit on main.
Git #02
Hard
🏢 Real Company
You accidentally committed your AWS secret key to GitHub. It's been there for 2 minutes. What do you do IMMEDIATELY?
✅ Critical incident response knowledge!
❌ Answer is A — REVOKE the key FIRST, always!
💡 Security Incident Response
Priority 1 (DO FIRST): AWS Console → IAM → Users → your user → Security credentials → Deactivate the leaked key. Even if it's only been seconds — bots scan GitHub in real-time.

Priority 2: Check AWS CloudTrail for any calls made with that key in the last 2 minutes.

Priority 3: Clean git history: git filter-branch --force --index-filter "git rm --cached --ignore-unmatch .env" HEAD OR use BFG Repo Cleaner (faster). Force push: git push --force

Note: Making repo private (Option C) helps but GitHub caches are searchable and bots may have already scraped it. Always treat a leaked key as COMPROMISED, not just hidden.
Monitor #01
Easy
📊 Golden Signals
Name the 4 Golden Signals of monitoring (from Google SRE Book) and briefly explain each.
✅ Perfect Answer — Memorize This!
1. Latency — How long requests take. Track p50, p95, p99 (not just average!). Alert if p99 > 2 seconds.

2. Traffic — How much demand on the system. Requests/second, WebSocket connections, messages/minute. Sudden drop = outage. Spike = viral event.

3. Errors — Rate of failed requests. HTTP 5xx rate. Alert if error rate > 0.1% for 5 minutes.

4. Saturation — How "full" the service is. CPU%, memory%, disk%, network bandwidth. At 100% = service degraded or down.

In DreamChat Grafana: 4 panels on the main dashboard showing all 4 signals. One glance = know if app is healthy.
Monitor #02
Medium
🔍 Prometheus
What is the difference between a Prometheus Counter and a Gauge? Give a real example of each.
✅ Metrics types mastered!
❌ Answer is C — Counter only goes up, Gauge can go both ways!
💡 Prometheus Metric Types
Counter — monotonically increasing. Never decreases (only resets to 0 on restart). Use for: total requests, total errors, total bytes sent. Query with rate() to get per-second rate.

Gauge — can go up or down. Snapshot of current state. Use for: active connections (goes up when user joins, down when they leave), memory usage, CPU%, queue depth.

DreamChat examples:
Counter: dreamchat_messages_total (total messages sent since start)
Gauge: dreamchat_active_connections (current WebSocket connections)
Design #01
Hard
🏛️ System Design
"Design a CI/CD pipeline for a team of 10 developers deploying a Node.js app to Kubernetes 5 times a day." Walk through your design.
✅ Senior-Level System Design Answer
Git Strategy: Feature branches → develop → main. Protected main with required PR reviews + CI pass.

CI Pipeline (GitHub Actions, triggers on every PR + push to main):
1. Run unit + integration tests
2. Static code analysis (SonarQube)
3. Dependency vulnerability scan (OWASP)
4. Docker build (multi-stage)
5. Container scan (Trivy) — block CRITICAL CVEs
6. Push to registry tagged with git SHA

CD Pipeline (ArgoCD GitOps):
CI updates deployment.yaml with new image tag → commits to Git → ArgoCD detects → deploys to staging auto → manual approval gate → deploy to prod

Rollback: Git revert → CI rebuilds → ArgoCD deploys previous image. Or: kubectl rollout undo deployment/app for instant pod-level rollback.

Monitoring: Prometheus alerts on error rate spike post-deploy → automatic rollback trigger via webhook.

🟢 Project 01
Dockerize a Static Website
Master: Docker basics · Dockerfile · nginx · image build · run · port mapping
🐳
#01
Your First Dockerized Website
Take a simple HTML/CSS site (or use a free template), write a Dockerfile, build the image, and serve it with nginx inside a container on your M4 Mac. Understand images vs containers vs volumes.
DockernginxHTML/CSS
⏱ 1–2 hours
Beginner
📋 Step-by-Step Guide — Click to Expand
▼
💡
What you need: Docker Desktop installed on M4 Mac. If not: go to docker.com/products/docker-desktop → Download for Mac (Apple Silicon) → install → open it.
1
Create your project folder
Open Terminal on your M4 Mac and run:
mkdir my-docker-site && cd my-docker-site
mkdir html
nano html/index.html
In nano, type a simple HTML page, then press Ctrl+X → Y → Enter to save.
2
Write the Dockerfile
In the my-docker-site folder, create a file called Dockerfile (no extension):
# Base image — nginx web server on Alpine Linux (tiny, 5MB)
FROM nginx:alpine

# Copy our HTML files into nginx's web root
COPY html/ /usr/share/nginx/html/

# Expose port 80 (nginx default)
EXPOSE 80
3
Build the Docker image
# Build image, tag it "mysite:v1"
docker build -t mysite:v1 .

# Verify image was created
docker images
You should see mysite in the list with tag v1.
4
Run the container
docker run -d -p 8080:80 --name my-website mysite:v1

# -d = run in background (detached)
# -p 8080:80 = map Mac port 8080 → container port 80
# --name = give it a friendly name
Open browser → http://localhost:8080 → you see your site! 🎉
5
Explore & learn essential Docker commands
docker ps # list running containers
docker ps -a # all containers (including stopped)
docker logs my-website # see nginx access logs
docker exec -it my-website sh # get a shell INSIDE the container
docker stop my-website # stop it
docker rm my-website # delete container
docker rmi mysite:v1 # delete image
🏆
Interview win: "I Dockerized a static site using nginx:alpine — kept image size under 10MB using multi-stage build best practices. Container ran with non-root user for security."

#02
Git Branching & Collaboration Lab
Create a repo, simulate a team (2 branches), create a merge conflict intentionally, resolve it, open a PR, review it, merge — the full professional workflow in one solo exercise.
GitGitHubCLI
⏱ 2 hours
Beginner
📋 Step-by-Step Guide — Click to Expand
▼
1
Create repo on GitHub
Go to github.com → top right + → New repository → name: git-lab → check Add README → Create repository. Then clone it:
git clone https://github.com/akshc15/git-lab.git
cd git-lab
2
Create two feature branches
# Branch 1 — feature A
git checkout -b feature/add-login
echo "Login page here" > login.txt
git add . && git commit -m "Add login page"
git push origin feature/add-login

# Branch 2 — feature B (from main)
git checkout main
git checkout -b feature/add-dashboard
echo "Dashboard here" > dashboard.txt
git add . && git commit -m "Add dashboard"
git push origin feature/add-dashboard
3
Create a Pull Request on GitHub
Go to github.com/akshc15/git-lab → you'll see a yellow banner "Compare & pull request" → click it → add a description → click Create pull request → then click Merge pull request → Confirm merge. Do this for both branches.
4
Create & resolve a merge conflict (on purpose!)
# Both branches edit the SAME line in README — conflict!
git checkout -b conflict-branch-a
echo "Version A" > README.md && git commit -am "A's version"
git push origin conflict-branch-a

git checkout main
git checkout -b conflict-branch-b
echo "Version B" > README.md && git commit -am "B's version"
git merge conflict-branch-a
# Now you'll see the conflict markers in README.md
# Edit the file, remove markers, keep what you want
git add README.md && git commit -m "Resolved conflict"

#03
Auto-Deploy Site with GitHub Actions
Write a GitHub Actions workflow that automatically builds and deploys your website to GitHub Pages every time you push to main. Your first real CI/CD pipeline — zero servers needed, 100% free.
GitHub ActionsYAMLCI/CDGitHub Pages
⏱ 2 hours
Beginner
📋 Step-by-Step Guide — Click to Expand
▼
1
Create the workflow file
In your repo, create this path: .github/workflows/deploy.yml
# Every push to main triggers this workflow
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest # free GitHub runner
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./ # deploy from root
2
Watch it run on GitHub
Push your workflow file → go to your repo → click Actions tab → you'll see your workflow running in real-time. Green checkmark = success. Click on the job to see each step's output. This is your pipeline dashboard.
3
Add a build step (level up)
Extend the workflow to also lint your HTML before deploying — this is the "CI" part (catch errors before deploy):
- name: Lint HTML
  run: |
    npx htmlhint *.html || true
    # "|| true" so pipeline doesn't fail on warnings
🏆
Interview win: "I built CI/CD with GitHub Actions — triggered on push, runs linting and tests, then deploys automatically. I understand jobs, steps, runners, secrets, and workflow artifacts."

#04
Automated Linux Server Setup Script
Spin up a free EC2 t2.micro on AWS, SSH in, write a Bash script that auto-installs + configures nginx, sets up a cron job, creates a systemd service, configures firewall (ufw), and logs server stats. Your first "Infrastructure as Script".
LinuxBashAWS EC2nginxsystemd
⏱ 3 hours
Beginner
📋 Step-by-Step Guide — Click to Expand
▼
1
Launch a free EC2 instance on AWS
Login to AWS Console → search EC2 → click Launch Instance:
1
Name: linux-lab-server
2
AMI: Ubuntu Server 22.04 LTS — click Select
3
Instance type: t2.micro (free tier eligible — look for the green "Free tier" label)
4
Key pair: click Create new key pair → name: linux-lab-key → RSA → .pem → Create. File downloads automatically.
5
Network settings: check Allow SSH from My IP and Allow HTTP traffic
6
Click Launch instance → wait 1 min → click View all instances → note the Public IPv4 address
2
SSH into your server from M4 Mac
# Secure the key file first
chmod 400 ~/Downloads/linux-lab-key.pem

# SSH in (replace IP with your EC2 public IP)
ssh -i ~/Downloads/linux-lab-key.pem ubuntu@YOUR-EC2-IP

# You're now inside your Linux server! Prompt changes to:
# ubuntu@ip-172-xx-xx-xx:~$
3
Write the setup bash script
nano setup.sh
Paste this script:
#!/bin/bash
# === SERVER SETUP AUTOMATION SCRIPT ===
set -e # Exit on any error

echo "[1/5] Updating system..."
sudo apt update -y && sudo apt upgrade -y

echo "[2/5] Installing nginx..."
sudo apt install nginx -y
sudo systemctl enable nginx
sudo systemctl start nginx

echo "[3/5] Configuring firewall..."
sudo ufw allow OpenSSH
sudo ufw allow 'Nginx HTTP'
sudo ufw --force enable

echo "[4/5] Creating server stats script..."
cat > /home/ubuntu/stats.sh << 'EOF'
#!/bin/bash
echo "=== Server Stats $(date) ===" >> /home/ubuntu/stats.log
echo "CPU: $(top -bn1 | grep 'Cpu(s)' | awk '{print $2}')%" >> /home/ubuntu/stats.log
echo "Memory: $(free -h | grep Mem | awk '{print $3"/"$2}')" >> /home/ubuntu/stats.log
echo "Disk: $(df -h / | tail -1 | awk '{print $5}')" >> /home/ubuntu/stats.log
EOF
chmod +x /home/ubuntu/stats.sh

echo "[5/5] Adding cron job (runs every 5 min)..."
(crontab -l 2>/dev/null; echo "*/5 * * * * /home/ubuntu/stats.sh") | crontab -

echo "✅ Setup complete! Visit: http://$(curl -s ifconfig.me)"
chmod +x setup.sh && ./setup.sh
4
Verify everything works
# Check nginx status
systemctl status nginx

# View server in browser
# Open: http://YOUR-EC2-IP → nginx default page

# Check cron is running
crontab -l

# After 5+ min, check stats log
cat ~/stats.log
💸
Save cost: When done practicing, stop (don't terminate) the instance: AWS Console → EC2 → select instance → Instance state → Stop instance. t2.micro stopped = free.

#05
WordPress + MySQL with Docker Compose
Deploy a full WordPress site with MySQL database using a single docker-compose.yml file. Learn how containers communicate on private networks, how volumes persist data, and how to manage secrets via environment variables. All on your M4 Mac — free, no cloud needed.
Docker ComposeMySQLWordPressVolumesNetworks
⏱ 2 hours
Beginner
📋 Step-by-Step Guide — Click to Expand
▼
# Create project folder
mkdir wp-compose && cd wp-compose
nano docker-compose.yml
version: '3.8'

services:
  db:
    image: mysql:8.0
    restart: always
    environment:
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wpuser
      MYSQL_PASSWORD: strongpassword123
      MYSQL_ROOT_PASSWORD: rootpassword123
    volumes:
      - db_data:/var/lib/mysql # persistent data!
    networks:
      - wp-network

  wordpress:
    image: wordpress:latest
    depends_on: [db] # start after db
    ports:
      - "8080:80"
    environment:
      WORDPRESS_DB_HOST: db:3306
      WORDPRESS_DB_USER: wpuser
      WORDPRESS_DB_PASSWORD: strongpassword123
      WORDPRESS_DB_NAME: wordpress
    networks:
      - wp-network

volumes:
  db_data: # named volume — persists between restarts

networks:
  wp-network: # private network — only these 2 containers talk
docker compose up -d # start everything
docker compose ps # see running services
docker compose logs -f # follow logs from all containers
# Open browser: http://localhost:8080 → WordPress setup!

docker compose down # stop and remove containers
docker compose down -v # also remove volumes (delete data)

🔵 Detailed Guide — Project 07
Ansible: Complete Walkthrough
Most confusing tool for beginners — extra detail here
📋 Ansible Step-by-Step — Click to Expand
▼
🧠
Mental model: Ansible reads a YAML "playbook" (your instructions), SSHs into target servers, and runs tasks. No agent needed on servers. You control it from your M4 Mac. It's idempotent — run it 10 times, same result.
1
Install Ansible on M4 Mac
brew install ansible
ansible --version # verify
2
Create inventory file (tells Ansible WHERE your servers are)
mkdir ansible-lab && cd ansible-lab
nano inventory.ini
[webservers]
server1 ansible_host=YOUR-EC2-IP-1 ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/linux-lab-key.pem
server2 ansible_host=YOUR-EC2-IP-2 ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/linux-lab-key.pem
# Test connectivity to all servers
ansible webservers -i inventory.ini -m ping
3
Write your first playbook
nano setup-webserver.yml
---
- name: Configure Web Servers
  hosts: webservers
  become: yes # sudo

  tasks:
    - name: Update apt cache
      apt: update_cache=yes

    - name: Install nginx
      apt:
        name: nginx
        state: present # idempotent: install if not there

    - name: Start and enable nginx
      service:
        name: nginx
        state: started
        enabled: yes

    - name: Create custom index page
      copy:
        content: "<h1>Managed by Ansible!</h1>"
        dest: /var/www/html/index.html
# Run the playbook!
ansible-playbook -i inventory.ini setup-webserver.yml

# -v, -vv, -vvv for more verbosity when debugging

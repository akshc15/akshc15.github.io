🚨 REAL ERRORS YOU'LL HIT + FIXES
COPY failed: file not found in build context or excluded by .dockerignore: stat package.json
You're running `docker build` from wrong directory! Run it from folder that contains Dockerfile:
cd my-app && docker build -t myapp .
The DOT (.) at end = "use current folder as context"
Error response from daemon: driver failed programming external connectivity: port is already allocated
Another process is using that port. Find and kill it:
lsof -i :8080 → shows what's using port 8080
kill -9 [PID] → kill it
OR use a different port: docker run -p 8081:8080 myapp
OCI runtime exec failed: exec: "sh": executable file not found in $PATH
You used Alpine Linux image which uses `ash` not `sh`. Use: docker exec -it container_id /bin/ash
OR switch FROM alpine to FROM ubuntu in Dockerfile

🚨 REAL K8s ERRORS + FIXES
NAME: frontend-pod-xxx STATUS: CrashLoopBackOff RESTARTS: 6
Container keeps crashing and K8s keeps restarting it.
Step 1: kubectl logs frontend-pod-xxx → Read the actual error message
Step 2: kubectl describe pod frontend-pod-xxx → Look at "Events" section at bottom
Common causes: App has a bug, wrong ENV vars, can't connect to DB, wrong image tag
0/2 nodes are available: 2 Insufficient memory. preemption: 0/2 nodes are available
Your pod wants more memory than nodes have free!
Fix 1: Lower memory request in deployment.yaml: resources.requests.memory: "128Mi"
Fix 2: Add more nodes to your EKS cluster with eksctl
eksctl scale nodegroup --cluster=NAME --nodes=3 -n nodegroup-name
ImagePullBackOff — Failed to pull image "123456.dkr.ecr.us-east-1.amazonaws.com/frontend:latest"
K8s can't pull your image from ECR. 3 possible causes:
1. Wrong image name — typo in deployment.yaml image field
2. No ECR secret — create kubectl create secret docker-registry ecr-secret ...
3. Node IAM role missing ECR permission — add AmazonEC2ContainerRegistryReadOnly policy to EKS node role in AWS Console

✍️ WITH TERRAFORM: CODE = INFRA
You write .tf files:

Terraform does the work:

Terraform is IDEMPOTENT — run apply 100 times, same result. It only changes what's different between your code and reality.
🌍 THE TERRAFORM STATE STORY
📸
Terraform's state file (terraform.tfstate) is like a PHOTO of everything it created.

Without it: "Did I already create that VPC? Where's my EC2? How many subnets do I have?" — CHAOS!

With it: Terraform knows exactly what exists and what needs to change.

In our project → State stored in S3 bucket. Why? Team members can share it. It has version history. No accidental duplicate creates.
Error: Error acquiring the state lock. Lock Info: ID: xxx Path: s3://bucket/terraform.tfstate
Someone else (or a crashed previous run) left a lock on the state. Only do this if you're SURE no one else is running Terraform!
terraform force-unlock LOCK_ID
Get LOCK_ID from the error message above.
Error: Provider "registry.terraform.io/hashicorp/aws" was not found. Run terraform init
You forgot to run terraform init first! Or you're in the wrong directory.
cd Jenkins-Server-TF/
terraform init
This downloads the AWS provider plugin into .terraform/ folder.

🔐 THE 3 SECURITY GUARDS
🔍
SonarQube (SAST)
Reads your SOURCE CODE like a smart reviewer.
"You have SQL injection risk on line 47!"
"This function is too complex!"
"You have hardcoded password!"
🛡️
OWASP Dep-Check
Checks your package.json like a safety inspector.
"You use lodash 4.17.15 which has CVE-2021-23337 — it can be exploited to run arbitrary commands!"
🔎
Trivy (Container Scan)
Scans the BUILT DOCKER IMAGE.
"Your Ubuntu base image has 12 CRITICAL OS vulnerabilities. Update: FROM ubuntu:22.04"
SAST = Static (source code scan). SCA = Dependencies (OWASP). CSPM = Container (Trivy). These 3 cover your entire attack surface.
🚨 JENKINS ERRORS + FIXES
ERROR: No such DSL method 'withAWS' found among steps
You're missing the "Pipeline: AWS Steps" Jenkins plugin!
Jenkins → Manage Jenkins → Plugins → Available → search "Pipeline AWS" → Install
Then restart Jenkins: sudo systemctl restart jenkins
hudson.plugins.git.GitException: Command "git fetch ..." returned status code 128: stderr: Permission denied (publickey)
Jenkins can't authenticate with your GitHub repo.
Fix: In Jenkins → Credentials → Add your GitHub Personal Access Token (NOT password!)
Generate token: GitHub → Settings → Developer Settings → Personal Access Tokens → check 'repo' scope
Then update your pipeline to use this credential ID.
Error response from daemon: denied: Your authorization token has expired. Reauthenticate and try again.
ECR login tokens expire every 12 hours. Your pipeline must re-login on every run:
Add this to your Jenkinsfile before docker push:
sh 'aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin $ECR_URL'
SonarQube: QUALITY GATE FAILED — Reliability Rating is D (expected A)
Not really an error — SonarQube found real bugs! Options:
1. Fix the bugs it found (correct approach)
2. In SonarQube dashboard → your project → Quality Gate → customize thresholds for learning
3. In Jenkinsfile add waitForQualityGate abortPipeline: false to continue even if gate fails (while learning)

🚨 EKS ERRORS + FIXES
error: You must be logged in to the server (Unauthorized) — kubectl get nodes
Your kubeconfig is outdated or wrong user.
aws eks update-kubeconfig --region us-east-1 --name three-tier-cluster
Also check: aws sts get-caller-identity — are you logged in as the RIGHT IAM user that created the cluster?
If cluster was created by a different role, you need to add your user to the aws-auth ConfigMap.
Error: The connection to the server localhost:8080 was refused
kubectl doesn't know which cluster to talk to. Your kubeconfig is empty or missing.
export KUBECONFIG=~/.kube/config
aws eks update-kubeconfig --region us-east-1 --name three-tier-cluster
kubectl cluster-info ← verify connection
EKS cluster creation failed: Cannot create cluster — VPC limit exceeded
AWS default limit is 5 VPCs per region. You may have hit it from previous experiments.
Go to AWS Console → VPC → delete old/unused VPCs
OR request a limit increase at AWS Service Quotas.
🌊 EPISODE 7: "THE GUARDIAN"
(ArgoCD + GitOps)
🌊 ARGOCD IS A TEMPLE GUARDIAN
🗿
ArgoCD sits inside your Kubernetes cluster, staring at your Git repo 24/7.

The moment you change a Kubernetes YAML file in GitHub, ArgoCD sees it and says:
"Cluster state doesn't match Git state. SYNCING NOW."

It automatically applies the changes. No human needed. This is GitOps.
Git = Single source of truth. NEVER run kubectl apply manually in GitOps. Always commit to Git and let ArgoCD apply it. This gives you full audit trail and easy rollback.
🔄 THE FULL GITOPS LOOP
1️⃣ Dev pushes code to GitHub → Jenkins runs CI pipeline
⬇
2️⃣ Jenkins builds Docker image → pushes to ECR → tags it :v1.5
⬇
3️⃣ Jenkins updates deployment.yaml in GitHub: image: .../frontend:v1.5
⬇
4️⃣ ArgoCD detects change in Git repo within 3 minutes
⬇
5️⃣ ArgoCD applies new deployment.yaml → K8s does rolling update → New pods running!
🚨 ARGOCD ERRORS + FIXES
App Status: OutOfSync — ComparisonError: failed to get git client: unknown key type: "ecdsa-sha2-nistp256"
ArgoCD can't authenticate with GitHub. Use HTTPS with token instead of SSH:
In ArgoCD UI → Settings → Repositories → + Connect Repo → HTTPS
URL: https://github.com/youruser/yourrepo
Username: your github username
Password: your GitHub PAT (Personal Access Token with repo scope)
App Status: Degraded — Replica Set not available: Minimum availability not yet met
Your pods are failing to start. Debug with:
kubectl get pods -n three-tier ← find failing pods
kubectl describe pod PODNAME -n three-tier ← see Events section
kubectl logs PODNAME -n three-tier ← see app logs
Usually: wrong image tag, missing secret, wrong env var
ArgoCD App Health: Progressing (stuck for 10+ minutes)
Deployment is stuck — new pods not becoming Ready.
kubectl get pods -n three-tier -w ← watch in real-time
Look for: Pending (no resources), ErrImagePull (bad image), CrashLoopBackOff (app crashes)
Fix the underlying pod issue → ArgoCD will automatically show Healthy once pods are running

😎 WITH HELM
⚓
Helm = apt-get / brew for Kubernetes

Instead of 15 YAML files, just:

helm repo add prometheus-community https://...
helm install prometheus prometheus-community/kube-prometheus-stack

Done! Prometheus + Grafana + AlertManager all running. 30 seconds. Zero manual YAML.
Helm Chart = A package of K8s YAML templates with configurable values
helm install = install a chart
helm upgrade = update it
helm rollback = go back to previous version
helm uninstall = remove everything cleanly
Chart.yaml = package metadata. values.yaml = default config. templates/ = the actual YAML files with {{ .Values.xxx }} placeholders.
🚨 HELM ERRORS + FIXES
Error: INSTALLATION FAILED: cannot re-use a name that is still in use: "prometheus"
You already installed a release with this name. Options:
helm list -n monitoring ← see existing releases
helm upgrade prometheus prometheus-community/kube-prometheus-stack -n monitoring ← upgrade instead
OR: helm uninstall prometheus -n monitoring then install again
Error: failed to download "prometheus-community/kube-prometheus-stack" — repo not found
You haven't added the Helm repo yet:
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm search repo prometheus-community ← verify it's added

🔍 WHAT TO WATCH — THE GOLDEN SIGNALS
⚡
Latency
How long do requests take? p99 > 2s = something is wrong
🚦
Error Rate
% of 5xx responses. > 1% = your app is broken for users
📈
Traffic
Requests per second. Sudden drop = outage. Spike = viral
💻
Saturation
CPU/Memory usage. > 80% = scale up or you'll crash
The 4 Golden Signals (Google SRE): Latency, Traffic, Errors, Saturation. Memorize these — you'll be asked in interviews!
🚨 PROMETHEUS + GRAFANA ERRORS + FIXES
Grafana: "No data" — Data source connected but dashboards show empty panels
Prometheus isn't scraping your targets. Check:
Port-forward Prometheus: kubectl port-forward svc/prometheus-kube-prometheus-prometheus 9090:9090 -n monitoring
Open http://localhost:9090 → Status → Targets
All targets should say "UP". If DOWN → your app's metrics endpoint isn't accessible.
kubectl port-forward: Unable to forward port because pod is not running
The Grafana/Prometheus pod crashed or is still starting.
kubectl get pods -n monitoring ← find the actual running pod name
kubectl logs PODNAME -n monitoring ← why it crashed
Common: Not enough memory on nodes. Add node or reduce resource requests.

🎯 THE ULTIMATE ERROR DEBUGGING FLOWCHART
❓ App not working?
→ kubectl get pods -n three-tier — what's the STATUS?
STATUS = Running but still broken → kubectl logs POD_NAME → app-level error → fix your code
STATUS = CrashLoopBackOff → kubectl logs POD_NAME --previous → app crashed → fix code/config
STATUS = Pending → kubectl describe pod POD_NAME → Events section → no resources / scheduling issue
STATUS = ErrImagePull → wrong image tag or ECR auth failed → check deployment.yaml image name + ECR secret
STATUS = OOMKilled → out of memory → increase resources.limits.memory in deployment.yaml
